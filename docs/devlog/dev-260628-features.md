## 2026-06-28 STEP2 传输性能与遗留问题处理

### 网络传输性能
- 优化 P2P DataChannel 背压等待：不再依赖可能延迟触发的 `bufferedamountlow` 事件和 1 秒兜底等待，改为事件加 20ms 轮询组合，降低同 WiFi 下大文件 P2P 传输被缓冲等待拖慢的概率。
- P2P 和 Socket.IO relay 单片保持 64KB，避免 iOS/Safari 系 DataChannel message size 兼容风险；主要通过扩大 P2P 缓冲窗口和修复等待策略提升吞吐。
- 强制拉取缓存时将目标文件加入优先队列，优先于普通后台补缓存任务调度。
- 进度 UI 更新增加 120ms 节流，减少大文件传输时频繁 DOM 更新拖慢纯文本/富文本同步的概率。

### 进度面板
- 传输进度面板默认缩小，减少多任务传输时遮挡主界面。
- 保留进行中的进度条置顶逻辑，上行和下行只要最近确实有进度增长都会优先显示。

### 磁链与做种
- 分享磁链时随请求携带本机缓存文件元数据，服务端确认设备在线后可直接注册本机为种子，降低 `File asset is not registered online` 的误报。
- 磁链缓存列表支持输入完整磁链地址或磁链 ID，先校验有效性，再在列表上方以内嵌 iframe 打开下载页；iframe 提供最小化和关闭按钮。

### 移动端音频
- 远端对讲/语音音轨到达时，在 iOS、微信内置浏览器、Opera 等环境主动显示声音启用按钮，并确保远端 audio 非静音、音量为 1。

### 富文本图片
- 富文本图片资源渲染前校验本地缓存 MIME，非 `image/*` 的缓存不再尝试按图片渲染，避免移动浏览器把异常二进制资源显示成乱码。

### 暂未完整改造
- 真正断点续传仍需要持久化 partial ranges/chunks 和服务端范围协商，本次只保留中断标记和优先重拉。
- 如果 iOS Opera 仍出现富文本图片乱码，需要继续采集具体 DOM 和缓存文件 type/size 日志定位。

## 2026-06-28 STEP2 续修 P2P 慢传与进度卡死

### P2P 背压与并发
- 修复 P2P DataChannel 背压等待可能卡在低水位以上但不触发超时的问题：如果缓冲长时间没有继续下降，会主动失败并进入重试/降级路径。
- 同一接收设备的大完整文件上传限制为单路，避免两个 20MB 文件同时争抢同一条 WebRTC/SCTP 通道；小文件仍允许插队并与大文件并行。

### 进度收尾
- P2P 发送中途失败并转 relay 时，主动收尾旧 P2P 发送进度，避免旧进度条停在某个百分比。
- 文件最终不可用时清理接收进度条，避免几百 KB 小图失败后遗留卡死进度。

## 2026-06-28 STEP2 传输尝试隔离
### P2P 与 Relay 重试隔离
- 为完整文件传输增加每次请求独立的 `attemptId`，P2P、Socket.IO relay、重试链路都会带上本次尝试标识，接收端只接收当前尝试的 chunk 和 complete，旧链路迟到的数据会被忽略。
- Socket.IO relay 服务端按 `attemptId` 单独登记完整文件中继状态，避免同一文件同一对设备的前后两次 relay 共享 `full` 状态而互相覆盖。
- P2P 发送端在发出 complete 后等待 DataChannel 缓冲排空，再把供源端进度标为 100%，减少“供源端已完成、接收端仍未落库”的错觉。
### 预期修复
- 针对 20MB 文件在 P2P/relay/重试之间反复切换时，接收端进度接近完成又回到 0、relay 100% 一闪而过但文件仍未缓存完成的问题，本次改动将不同尝试的接收缓冲隔离，避免旧尝试污染新尝试。

## 2026-06-28 STEP2 多源分片尝试隔离
### 分片级 Attempt
- 20MB 以上文件的多源分片现在也记录 `attemptId`，每个分片的 P2P/relay/重试都会隔离接收缓冲，旧分片链路迟到的 chunk 或 complete 不再写入当前分片。
- 服务端 relay 内部 key 从单纯 `transferId` 扩展为 `transferId + attemptId`，避免同一个 `part-N` 重试时服务端把两次中继流量记到同一笔 relay 状态。
- DataChannel 关闭时会校验 channel 所属 attempt 是否仍是当前分片 attempt，旧 channel 关闭不会触发当前分片重试，减少“快满后又回到 0”的误判。

## 2026-06-29 STEP2 验收问题修复
### 历史快照与设备发现
- 客户端加入隧道后会主动请求 `session-history` 快照，并在 3 秒、12 秒做兜底补拉，避免只依赖 join 时一次性推送导致新设备长时间没有传输记录。
- 老设备收到新设备加入事件后，会主动向服务端 reconcile 本地最近历史；即使服务端内存历史为空，新设备也能通过在线老设备恢复传输记录框架。
- 增加轻量 `tunnel-heartbeat`，服务端在心跳时刷新设备在线信息并回送全量在线设备列表；客户端以服务端列表收敛本地设备列表，减少各设备看到在线人数不一致。
### 消息可靠性
- 服务端收到文本/富文本/文件消息后向发送端回 `message-ack`，客户端 5 秒未收到 ACK 会主动请求历史快照补偿。
- `addMessageToChat` 和实时消息处理改为按 `messageId` 幂等渲染，避免高负载下实时消息与快照撞车导致页面临时出现重复消息。
### 强制拉取与进度 UI
- 强制拉取缓存的文件保持优先队列，并允许临时多开一个下载槽位，降低被普通后台补缓存任务压住的概率。
- 传输进度文件名左侧增加方向图标：红色上箭头表示上传，绿色下箭头表示下载，原状态文案不变。
### 磁链缓存列表
- 修正缓存列表中内嵌 downloader iframe 的宽度约束，避免 iframe 超出容器导致右上角最小化/关闭按钮不可见。
