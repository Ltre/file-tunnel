## 2026-06-29 History Window, Progress Anchor, Exit Actions

### 传输历史同步窗口
- 当前历史缺失优先不是 IndexedDB 上限导致，而是业务代码限制：服务端 `MAX_HISTORY_MESSAGES` 原先只有 100，客户端 `history-reconcile` 也只提交本地最后 100 条。
- 本次将服务端历史窗口提升到 1000 条、元数据窗口提升到 16MB；客户端 reconcile 同步窗口同步提升到 1000 条。
- 这仍然只同步传输记录元数据，不把大文件内容塞进历史快照；文件内容仍通过文件资产缓存链路拉取。
- 后续更适合继续做“时间线虚拟滚动 + 后台补齐缺口 + 顶部向上加载旧记录”的体验，而不是传统页码分页。

### 进度条定位
- 传输进度条现在记录基础文件 ID。
- 点击进度条会切到传输记录区域，并滚动定位到对应文件记录锚点。
- 定位成功后文件气泡会短暂高亮；如果对应记录尚未渲染，会提示稍后再试并写入 debug log。

### 断点传输状态
- 当前代码有 `transferInterrupted/isPartial` 等中断标记，也有重试机制，但还没有真正持久化已接收分片并从断点续传。
- 刷新页面、断线、突然离线后，大文件仍可能重新拉取完整文件或重新拉取分片；完整断点续传仍是后续架构项。

### 连接设备区域底部操作
- 在“连接设备”区域底部新增“暂时离开”按钮，保留顶部同类按钮。
- 底部“暂时离开”和“退出隧道”采用 2:1 横向布局，并用 45 度斜切轮廓形成一组动作条。

## 2026-06-30 Short Code, Manifest, Transfer Architecture Notes

### 短码输入
- 修复 Windows Chrome 在隧道暗号方框输入单个字符时，字符被重复推进到下一个方框的问题。
- 第一格不再使用 `autocomplete="one-time-code"`，避免桌面浏览器走奇怪的自动填充路径。
- 对单字符键入使用 `beforeinput` 接管：先阻止浏览器默认写入，再由代码写入当前格并移动焦点；粘贴整串短码仍走原有分发逻辑。

### 动态 Manifest
- `/manifest.webmanifest` 改为服务端动态输出，并且路由位于静态文件服务之前。
- 新增 `manifest.hosts.json`，支持按 Host 映射完整 manifest JSON。
- 当前配置包含：
  - `tun.miku.us` -> `即时传输隧道(韩大狗)` / `传输隧道(韩大狗)`
  - `tun-txhk.miku.us` -> `即时传输隧道(疼HK)` / `传输隧道(疼HK)`
  - `tun-alyhk.miku.us` -> `即时传输隧道(啊HK)` / `传输隧道(啊HK)`
  - `default` -> `即时传输隧道(啊HK)` / `传输隧道(啊HK)`

### 垃圾清理确认
- 已审查“清理垃圾”逻辑：当前扫描会命中未被消息、富文本、协同编辑引用的文件缓存，也会命中 `isPartial` 或 `transferInterrupted` 标记的中断传输缓存。
- 当前不额外修改删除策略，避免把仍可恢复的正常文件误判为垃圾。

### 需要拆分实现的传输/数据模型需求
- HTTP Range Relay 和真正断点续传需要统一设计：目前已有多源 range 传输与 socket relay range，但还没有“服务端 HTTP range 临时中继 + 浏览器本地分片持久化 + 断点恢复协商”的完整协议。
- 合辑消息、跨隧道/跨设备转发、合辑预览、以及把“详情/下载/分享磁链/清除缓存/删除”迁移到文件预览浮层，都会改变消息数据模型和现有文件交互入口，后续应按数据结构、渲染、缓存引用、删除语义分阶段实现。
- 隧道备注名需要服务端 session 元数据、客户端本地 sessions 缓存、加入会话 snapshot、以及在线广播协同更新；建议接入 infra DB 后再持久化，避免服务器重启丢备注。
## 2026-06-30 Collection Messages

### 合辑发送入口
- 多选或拖拽多个文件时，先弹出发送方式选择层：可以按合辑发送，也可以拆分成多条普通文件记录发送。
- 合辑发送时仍然为每个文件创建独立文件资产缓存，复用现有 blob/IndexedDB/P2P 拉取链路；历史记录层只新增一条 `type: "collection"` 消息，保存文件列表、数量和总大小。

### 合辑预览与打开
- 传输记录中的合辑气泡采用 2x2 堆叠预览，超过 4 个文件时在最后一个预览格展示“更多文件...”和剩余数量。
- 点击合辑气泡后进入合辑文件宫格；点击宫格中的单个文件，再进入原有文件预览浮层。
- 合辑文件缓存完成或清理后，会刷新合辑预览缩略图，避免合辑仍显示过期缩略图。

### 文件动作按钮迁移
- 单文件记录下方不再直接显示“详情 / 下载 / 分享磁链 / 清除缓存 / 删除”按钮。
- 单文件点击打开预览浮层后，在浮层底部显示这些动作按钮。
- 合辑打开的第一层只显示“下载全部”批量动作；进入合辑中的单个文件预览后，才显示单文件动作按钮。

### 辅助逻辑适配
- 历史同步、实时消息、snapshot duplicate 分支都补充了合辑文件缓存拉取。
- 垃圾清理、按文件 ID 查历史记录、传输进度条定位锚点均识别合辑内文件，避免把合辑文件误判成游离缓存或找不到锚点。

## 2026-06-30 Collection Acceptance Fixes

### 历史顺序与同步
- 传输记录渲染按 `timestamp + localOrder + messageId` 插入，减少多设备实时到达顺序抖动导致的列表乱序。
- authoritative `session-history` 快照现在会对比同 ID 消息内容；如果服务端合辑已更新而本地仍是旧版本，会应用 `message-updated` 等价更新，避免错过实时事件后只能靠手动刷新才对齐。
- `history-reconcile` 会剥离合辑文件里的 inline data，保持同步负载和服务端历史消息格式一致。

### 预览浮层与文件动作
- 文件预览浮层改为纵向 flex 布局，内容区独立滚动，底部操作按钮不再被超高图片顶出容器。
- 文件详情浮层层级高于文件预览浮层，避免“详情”被预览层遮挡。
- 无本机缓存和不可直接预览的文件也会打开文件操作浮层，中间显示文件元信息，仍可执行详情、下载、分享链接、清除缓存、删除等动作。
- 单文件、普通文本、富文本和合辑普通视图层补充轻量删除入口。

### 合辑内文件语义
- 合辑内单文件删除改为更新合辑文件列表，不再误删整条合辑。
- 新增 `update-message` / `message-updated` 同步路径；服务端更新合辑后会广播消息更新并再次广播会话快照。
- 收到合辑更新时，会清理被移除且不再被其它文件消息、合辑、富文本或编辑器内容引用的本地缓存。
- 点击合辑内单文件打开的文件层关闭时，只返回合辑宫格层，不再连同合辑层一起关闭。
- 合辑宫格中的不可预览/无缓存文件使用元信息卡片展示，并可继续打开单文件操作层。

### 移动端样式
- 合辑气泡和预览宫格收紧宽度约束，降低 Android Chrome 下右侧溢出的概率。
- 小屏幕下预览操作按钮自动两列伸展，避免横向挤出。

## 2026-06-30 Collection Acceptance Fixes v3

### 合辑子文件缓存状态
- 合辑内单文件清除缓存后，文件操作层底部动作会从“清除缓存”切换为“还原文件”；还原请求中显示“正在还原”。
- 下载单个缺缓存文件时，只请求当前文件，并取消同合辑内已被用户清理且未主动还原的其它文件拉取队列，避免无关文件被自动拉缓存。
- 文件缓存拉取完成后，如果当前仍停留在该文件预览层，会自动从元信息状态刷新为图片/视频/音频/文档预览或可下载状态。

### 合辑宫格返回体验
- 从合辑宫格进入单文件预览前，会暂存合辑宫格 DOM、滚动位置和当前文件锚点。
- 关闭单文件预览时优先直接恢复暂存的合辑视图，不再先重建整个宫格，降低大量图片/视频时关闭卡顿。
- 返回合辑宫格后会定位到刚才打开的文件卡片，并短暂高亮。

### 合辑下载全部
- “下载全部”改为先拉取缺失缓存，等待文件全部就绪后统一生成 ZIP 下载。
- 等待层显示当前已就绪数量，并提供“不等了，先下载再说”按钮，可打包当前已缓存文件形成不完整 ZIP。
- 避免浏览器连续触发多个单文件下载导致实际下载数量少于提示数量。

### 资源浏览器引用
- 资源浏览器现在把合辑内文件识别为 `collection-file` 引用，避免仍在合辑中的文件被标记为“未引用”。
- 合辑文件引用可定位回传输记录中的合辑消息锚点。

## 2026-06-30 Collection Acceptance Fixes v4

### 被动打开与显式还原
- 合辑宫格点击子文件只打开单文件操作层，不再自动请求拉取缓存；只有“还原文件”“下载”“下载全部”等显式动作才会触发恢复。
- 单文件缺缓存预览也改为默认只展示元信息，避免用户刚清理缓存后因为查看详情又被自动拉回。
- 手动还原/下载会先发 provider discovery，再强制踢一次传输请求，用于处理旧请求卡死、服务端 provider 记录陈旧、点击后没有进度条的情况。

### 合辑远端更新增量化
- 收到远端合辑更新时，传输记录普通视图只刷新该合辑气泡，已打开的合辑宫格不再整格重建。
- 如果远端删除合辑中的某个文件，已打开的合辑宫格只移除对应卡片，并保留当前滚动位置。
- 如果用户正停留在被删除文件的单文件层，会先返回合辑层，再在合辑层中移除该文件卡片。

## 2026-07-01 Collection Preview Stack Fixes v5

### 预览容器稳定性
- 进入合辑子文件预览时会先立即切换标题并占位，避免大视频/大图片读取前短暂显示合辑标题。
- 图片/视频预览层使用固定的视口自适应高度；媒体内容按比例限制在容器高度的 90% 内并居中显示，单文件消息预览同样适用。

### 返回栈
- 合辑层 F 与子文件层 G 使用两级浏览器历史栈，Android Chrome 返回手势会先从 G 回到 F，再从 F 回到传输记录 P。

### 全屏媒体预览
- 可预览的图片/视频在 G 顶栏显示全屏按钮，进入全屏后可用关闭按钮、返回手势或 ESC 退出。
- 全屏模式支持左右箭头、键盘左右键和横向滑动在同合辑已缓存图片/视频之间切换。

## 2026-07-01 Collection Preview Media Fixes v6-from-v5

### 严格限定预览类型
- 传输记录文件预览只允许图片、视频、音频进入网页内预览。
- 文本、CSV、PDF、JSON、XML 等文件统一显示文件元信息和操作按钮，不再读取文件内容或创建 iframe/pre 预览，避免大文本文件卡死页面。

### 媒体尺寸约束
- 合辑子文件预览和单文件预览继续使用固定视口自适应容器。
- 图片/视频按容器高度 90% 显示，宽度按自身宽高比自动计算，避免竖屏视频/图片被底部截掉。

### 单播放器全屏
- 从 G 层进入全屏时，当前图片/视频 DOM 会移动到全屏层复用，不再新建第二个视频播放器。
- 退出全屏时把同一个媒体元素放回 G 层，保留视频播放进度和播放器状态。
- 切换到其它全屏文件时，会先把原 G 层媒体放回并暂停，保证页面上不会同时播放两个相同视频。

### 合辑宫格轻量化
- P 层合辑预览和 F 层合辑宫格不再为视频文件创建 video 标签，也不再为视频缩略图提前生成 Blob URL。
- 视频/音频文件在宫格中使用轻量图标和状态文案，只有点击进入 G 层时才真正加载媒体。
