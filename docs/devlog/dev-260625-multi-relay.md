# dev-260625-multi-relay 变更记录

## 2026-06-25 磁力下载、传输队列与管理能力补充

### 一、磁力下载页

- 磁力下载页在读取磁链后，会先检查 `TunnelDownloaderDB` 中是否已有完整磁力缓存；命中后直接显示“已命中本地缓存，可直接下载”，不再重新拉取。
- 如果磁力缓存未命中，会继续检查当前浏览器的隧道缓存 `TunnelDB.files`。当资源 ID 和磁链所属隧道一致，且本机已有完整文件内容时，进入“本地快速提取中...”状态，将该文件复制为磁力缓存副本，然后显示“已缓存到本机”。
- `/downloader` 页不再让“缓存列表”按钮直接跳转到 `/downloadList`，改为在当前页面打开 iframe 浮层，避免丢失磁链参数后出现“缺少磁链 ID”。
- 下载页会定时刷新 `/api/magnets/:id`，动态更新在线种子数量；如果下载仍在进行，会把新发现的种子设备交给传输层，以便大文件多源分片获得新的候选来源。

### 二、磁力缓存列表

- `/downloadList` 页面继续保留独立访问能力，同时支持被 `/downloader` 作为 iframe 嵌入。
- 每条磁力缓存记录提供“下载 / 复制磁链 / 删除缓存”三个操作。
- 由于浏览器无法可靠确认用户是否真正保存了 `<a download>` 触发的文件，下载后不会自动删除缓存，用户可在缓存列表中手动删除。

### 三、首页入口与设备列表

- 功能首页顶部固定条左起第二个按钮新增磁力缓存入口，点击打开 `/downloadList`。
- 同会话设备列表中，名为“磁链下载器”的临时下载设备不再显示一对一“对讲机”按钮，避免误操作。
- 首页顶部固定条空白处连续点击 7 次会打开 `/admin`，点击刷新、磁力缓存、暂时离开等按钮本身不会计数。

### 四、文件传输与多源分片

- 传输层将“完整文件拉取”和“大文件多源分片拉取”分开限流：完整文件最多 2 个并发，大文件多源任务额外最多 4 个并发。
- 多源分片仍保持每个大文件内部最多 4 个分片并发，并允许新发现的种子设备加入候选 provider 列表。
- 清除文件缓存或删除传输记录时，会调用传输模块的取消逻辑，并清理该文件相关的所有进度条。
- 传输模块新增取消标记，正在发送的 P2P DataChannel 或 Socket.IO relay 循环会在每个 chunk 前检查取消状态，防止文件已清理后发送进度继续增长形成“僵尸传输流”。

### 五、垃圾清理

- “清理垃圾”按钮点击后立即弹出扫描中的浮层，再异步扫描 IndexedDB，避免低配设备上用户等待数秒却没有任何反馈。

### 六、Admin 管理页

- 服务端新增 `/api/magnets`，用于返回当前服务器内存中已创建的全局磁力链接列表。
- 磁力链接是在传输记录区域点击“分享磁链”时创建的，不是文件发送时提前分配。
- Admin 页面新增“已分享磁链”面板，显示文件名、大小、类型、所属隧道、分享设备、创建时间、在线种子数，并提供打开和复制磁链操作。
- 已有“所有访问设备”列表继续显示设备 ID、IP、最后访问时间、在线/离线、活跃/不活跃等信息。

### 七、隧道切换入口

- 功能首页底栏“隧道”按钮支持桌面端右键、移动端长按，弹出本设备加入过的隧道列表，点击后可进入对应隧道。
- “多个隧道 iframe 同时常驻以保持多隧道活跃”的方案暂未直接实现，因为当前前端全局状态、Socket、IndexedDB 读写、媒体控制和 service worker 共享同一页面上下文，强行 iframe 常驻会带来较大的状态隔离风险。后续如果继续推进，建议将单隧道运行时拆成可多实例加载的子应用，再由外层 shell 管理多个 iframe。

### 八、后续大功能规划

- 传输记录区域“默认定位到底部或上次锚点，并优先加载锚点附近数据，再后台静默加载更多”的需求，当前仍保留现有全量本地渲染和底部定位逻辑。真正的 Telegram 式窗口化历史列表需要重构消息渲染、资源水合、滚动锚点和 DOM 回收策略，应作为单独任务推进。
- 联系人系统、设备 profile 二维码、不依赖隧道的联系人语音通话、联系人对讲机等能力，会涉及跨隧道身份、呼叫信令、权限确认、在线状态维护和持久联系人存储，建议拆分为独立联系人模块开发。
- Google Authenticator 管理页身份校验流程已整理到 `USAGE_GUIDE.md`，正式实现时需要增加 TOTP secret 生成、隐藏标记文件、管理会话 Cookie 和高风险管理 API 鉴权。
