## 1. 公众版最重要：先防滥用，再谈广告

你的产品本质上是“匿名即时文件传输 + 临时会话 + PWA”。这类工具很容易被拿来做：

* 传播恶意文件；
* 传播违规内容；
* 利用你的中继带宽刷流量；
* 通过短码爆破撞会话；
* 用作临时钓鱼/分发工具；
* 大文件刷满浏览器缓存或服务器中继资源。

所以公众版必须做成“**临时传输工具**”，而不是“匿名网盘”。

我建议公众版默认规则：

```text
未登录用户：
- 单文件大小限制，例如 200MB～1GB
- 单会话总量限制，例如 2GB
- 会话有效期，例如 24小时
- 短码有效期，例如 30分钟～2小时
- 空会话自动回收
- 中继流量限速
- 同 IP / 同设备会话数限制
- 禁止公开索引、禁止搜索引擎收录传输页
```

如果你保留 P2P 优先、Socket.IO/TURN 兜底，公众版要非常重视**中继成本**。P2P 成功时服务器成本低；一旦大量用户落到中继，服务器就是实打实承担上传+下载流量，10GB 文件可能变成约 20GB 服务器出站成本。

## 2. 文件安全：不要相信浏览器传来的文件信息

即使你主打 P2P，不把文件长期存服务器，也仍然要按“文件上传产品”的安全标准来设计。OWASP 的文件上传建议包括：限制扩展名、不要信任 `Content-Type`、生成服务端文件名、限制文件名长度、限制文件大小、只允许授权用户上传等。([cheatsheetseries.owasp.org][2])

公众版建议：

```text
1. 文件名只作为展示，不作为真实存储名
2. 所有下载默认走 attachment，不在站内直接执行/渲染
3. 不直接打开 HTML/SVG/JS/XML 等高风险类型
4. 图片预览只使用 Blob URL，不把用户内容写进 innerHTML
5. 富文本内容必须严格 sanitize
6. 任何用户输入都不要进入未转义 HTML
7. 禁止把文件内容、短码、sessionId 完整打进公开日志
8. 大文件分块必须校验顺序、大小、总长度、hash
```

尤其要注意 SVG、HTML、PDF、Office 文档这类文件。图片看起来无害，但 SVG 可以携带脚本，PDF/Office 也可能包含宏或漏洞利用链。公众版里可以允许传输，但不要在你的页面里主动内嵌执行。

## 3. 短码不能当唯一安全凭证

你现在有 5 位“隧道暗号”，这个适合用户体验，但不适合作为唯一访问凭证。

如果是 5 位数字+字母，理论空间是：

```text
36^5 = 60,466,176
```

看起来不少，但对公开服务来说，仍然需要防爆破。建议：

```text
1. 短码只用于“发现会话”，不是唯一权限
2. 真正会话 ID 继续用高熵 UUID/hash
3. 短码设置短有效期
4. 同 IP 输入错误短码限速
5. 多次错误后要求验证码/等待
6. 短码不要泄露完整会话历史
7. 进入隧道后最好还需要“发送端确认”或“房主批准”
```

更安全的设计是：

```text
短码 = 临时门牌号
session secret = 真正钥匙
```

也就是说，短码帮助找到会话，但不要让任何人只靠短码长期访问历史记录和资源。

## 4. WebSocket / Socket.IO 要做强约束

你的产品大量依赖 Socket.IO / WebRTC 信令 / 文件中继。WebSocket 类系统要特别注意鉴权、Origin 校验、限流、消息大小限制和日志审计；OWASP WebSocket 安全建议里也强调了连接建立/终止、鉴权、授权、限速触发、安全违规和异常断开的日志记录，同时避免记录敏感数据。([cheatsheetseries.owasp.org][3])

建议服务端所有 socket event 都加统一网关：

```text
- 校验 sessionId 格式
- 校验 deviceId 格式
- 校验 shortCode 格式
- 校验消息大小
- 校验 event 频率
- 校验设备是否在该 session
- 校验文件分块范围
- 校验 provider 是否真的声明过资源
- 校验 receiver 是否真的请求过资源
```

不要让客户端随便发：

```text
file-asset-chunk
file-asset-complete
history-reconcile
media-signal
delete-message
```

这类敏感事件。它们都应该带权限检查。

## 5. 公众版建议做“端到端加密”卖点

如果你要开放给公众，最能建立信任的卖点是：

> 文件在发送端浏览器加密，接收端浏览器解密，服务器即使中继也看不到明文。

现在你如果用 Socket.IO 中继分块，服务器理论上能接触到传输字节。即便你不落盘，也会带来隐私和合规压力。

建议做两种模式：

```text
普通模式：
- P2P/WebRTC 优先
- 中继只传输临时分块
- 适合日常使用

隐私模式：
- 发送端用 WebCrypto 生成随机 AES-GCM key
- 文件分块前先加密
- key 只通过 invite link / QR / WebRTC 安全通道传给接收端
- 服务器只看到密文
```

隐私模式可以作为商业化点：

```text
免费版：普通临时传输
Pro版：端到端加密、较大文件、更长会话、无广告
私有版：强制端到端加密、审计、内网部署
```

## 6. 广告可以做，但不要贴在“用户文件内容页”

你想植入广告是可以的，但要小心。Google Publisher Policies 明确说，放 Google 广告代码时，页面或 App 上的内容都要符合政策，内容包括用户生成内容、评论、其它广告和外部链接；违规内容变现可能导致广告被屏蔽、账号暂停或终止。([Google 支援][4])

所以广告位置要避开“用户文件内容”和“未审核富文本内容”。建议广告只放这些位置：

```text
1. 首页空状态
2. 创建隧道前
3. 传输完成后的工具推荐区域
4. 非会话内容页
5. 帮助文档页
6. 免费版 PWA 顶部/底部轻广告位
```

不建议放在：

```text
1. 用户富文本内容旁边
2. 文件详情弹层内
3. 图片/视频预览旁边
4. 共享会话的公开内容区域
5. 下载按钮旁边伪装成下载入口
```

广告不要影响传输主流程。这个产品如果广告太扰民，用户会很快转向 LocalSend、PairDrop、Snapdrop、微信/QQ 文件传输这类替代方案。

## 8. 私有部署版一定要产品化，不要靠手工改代码

你以后帮客户部署，不要每个客户手工 SSH 改一套。建议做标准包：

```text

```

```text

  ↓ HTTPS/WSS

  ↓
```


```text
Nginx + Node.js 集群 + Redis + PostgreSQL + MinIO + coturn 集群
```

私有部署版必须提供：


## 9. PWA 要继续做，但不要把 PWA 当成安全边界

PWA 对你这个项目很适合：用户可以安装到桌面或手机，分享文件时也可能作为入口。PWA 需要 manifest 来告诉浏览器应用的名称、图标、外观和行为；MDN 和 web.dev 都强调 manifest 是 PWA 可安装体验的重要组成。([MDN Web Docs][5])

但 PWA 也会带来几个坑：

```text
1. Service Worker 缓存旧 JS，导致线上 bug 难复现
2. Android 不同浏览器 PWA 行为不一致
3. Samsung Internet / Chrome / Edge 兼容性不同
4. 文件分享目标能力有限
5. 摄像头/麦克风仍然受 HTTPS、权限和设备存在性约束
```

你已经做了“强制刷新”按钮，这是对的。后续建议再加：

```text
- 当前前端版本号
- 当前 Service Worker 版本号
- 一键清理应用壳缓存
- 一键导出本地诊断日志
- 检测是否运行在 PWA 模式
- 检测是否 HTTPS
- 检测 mediaDevices / clipboard / File System Access 支持情况
```

做一个 `/debug` 或隐藏诊断页，非常有用。

## 10. “开放给公众”之前至少要补这几个后台能力

你现在的重点不是前端 UI，而是运营后台。

最低限度要有：

```text
1. 在线会话数量
2. 当前连接设备数量
3. 今日传输文件数
4. 今日中继流量
5. P2P 成功率
6. Socket.IO 中继占比
7. TURN 中继占比
8. 平均文件大小
9. 异常断开率
10. 错误日志聚合
11. IP / 设备 / session 限速
12. 封禁 IP / 短码 / 设备指纹
```

如果没有这些，公开后你不知道是用户正常增长，还是有人在刷你的中继。

尤其要做成本监控：

```text
今日出站流量
本月预计带宽账单
中继流量 Top IP
中继流量 Top session
大文件传输 Top session
```

一旦超过阈值自动降级：

```text
- 新建会话限速
- 免费用户禁止中继大文件
- 降低并发
- 要求登录
- 要求验证码
```

## 18. 最现实的商业路径

我觉得你这个项目不要一开始指望广告赚钱。更现实的是：

```text
公众免费版做流量入口
↓
技术用户觉得好用
↓
引导到“私有部署 / 团队版 / 去广告 Pro”
↓
靠部署、维护、定制收费
```

广告可以有，但只是补贴服务器成本。

真正能赚一点钱的是：

```text
“我帮你公司/团队部署一个自己的传输隧道”
```

你可以做一个价格页：

```text
免费版：公开站点，临时传输，有限制
Pro版：个人增强，无广告，更大传输
团队版：成员、审计、长会话
私有部署版：专属域名/内网部署/维护支持
```

对你现在的技术栈和项目成熟度来说，**先做私有部署能力，比先做广告系统更有商业价值**。
