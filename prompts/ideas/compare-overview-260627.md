竞品对比

--------------------------------------------------------------

发现产品优点是只要两台设备加入同一个隧道，不管何时何地，即便没传输完毕，只要某天双方同时在线，就能续传同一隧道那一批数据。

pairdrop没有做数据传输任务挂起后，持续保障续传和后续管理的功能，而file-tunnel就做了这么个临时容器确保用户对挂起的传输数据有掌控感。

一个很明显的优点就是一旦双方建立起共同的隧道，只要一方没有删除挂起的数据传输任务，随时都可以恢复数据传输任务，而且还能恢复一批任务





对，这个判断很关键。你现在抓到的不是一个“小功能优点”，而是 **file-tunnel 真正可以和 PairDrop / AirDrop / LocalSend 拉开差异的产品核心**。

它不只是“传文件”，而是：

> **建立一个临时但可持续的传输容器，让一批数据任务即使中断、离线、换网络，也仍然被用户看见、管理、恢复。**

这比单纯强调 P2P、Socket.IO、WebRTC 更有产品感。

## 这个定位比“即时传输”更强

PairDrop 这类产品更像：

> 两台设备临时相遇 → 立刻发一个文件 → 成功或失败 → 结束。

file-tunnel 可以变成：

> 两台设备加入同一个隧道 → 形成一批可追踪的传输任务 → 没传完也不会消失 → 以后双方再次在线自动/手动恢复 → 用户始终能看到这一批任务还在。

这就从“传输工具”升级成了 **传输任务容器**。

这个容器的价值是：用户不用记住“上次哪些文件没传完”“哪台设备还没收完”“传到多少了”“来源设备什么时候再在线”。这些状态都由隧道保存和呈现。

## 你应该把它包装成核心卖点

不要只说：

> 支持断点续传。

这太技术，别人听起来像下载器功能。

应该说：

> 隧道会保留未完成任务。只要双方以后再次同时在线，同一批文件可以继续恢复传输。

或者更产品化：

> 不怕断线，不怕换网络，不怕今天没传完。隧道会记住这批任务。

再进一步：

> 一次建立隧道，后续反复恢复。传输不是一次性动作，而是一组可管理任务。

这就是 file-tunnel 的差异点。

## 这个功能可以定义成“挂起传输任务”

建议你在 UI 和代码概念上明确引入：

```text
挂起任务
待恢复任务
未完成批次
可恢复资源
隧道任务队列
```

每个隧道不只是聊天记录和文件记录，而应该有一个非常显眼的区域：

```text
未完成传输 3 个
等待来源设备上线 2 个
可立即恢复 1 个
已完成 12 个
```

用户进入隧道后，第一眼就知道：

* 哪些文件已经完成；
* 哪些文件只传了一部分；
* 哪些文件来源设备当前离线；
* 哪些文件来源设备已经上线，可以恢复；
* 是否要一键恢复全部。

## 最值得做的 UI 形态

我建议在隧道里加一个 **任务抽屉 / 恢复中心**：

```text
传输任务

[可恢复] video-001.mp4
来源：电脑
目标：手机
进度：43%
状态：双方在线，可继续
操作：继续 / 删除 / 查看详情

[等待来源设备] backup.zip
来源：电脑
目标：手机
进度：18%
状态：电脑未在线
操作：提醒我 / 删除 / 保留任务

[已完成] photos.zip
来源：手机
目标：电脑
状态：已完成
操作：下载 / 清除缓存 / 删除记录
```

这样用户会有一种很强的掌控感：**东西没有丢，任务还在那里。**

这是普通“面对面传文件”工具很难给到的。

## 产品文案可以这样改

原来的定位：

> 即时传输隧道：安全、快速、去中心化的设备间文件传输工具。

可以改成：

> 一个可恢复的跨设备传输隧道。文件没传完也不会丢，双方以后再次在线即可继续。

更强一点：

> 为跨设备文件传输建立一个临时任务容器。断线、换网、离线后，任务仍可保留和恢复。

再口语化：

> 今天没传完，明天接着传。

这个很有记忆点。

## 技术上要把“续传”做成显性资产

现在你已经有资源浏览器、本地缓存、provider、restore、file asset request、多源分片这些基础。下一步应该把它们产品化成一个明确模型：

```js
TransferTask {
  id,
  tunnelId,
  batchId,
  assetId,
  sourceDeviceId,
  targetDeviceId,
  fileName,
  fileSize,
  receivedBytes,
  status: 'pending' | 'transferring' | 'paused' | 'waiting-source' | 'recoverable' | 'completed' | 'failed',
  lastRoute,
  lastProgressAt,
  createdAt,
  updatedAt
}
```

特别是要有 `batchId`。因为你说得对，优势不只是恢复一个文件，而是**恢复一批任务**。

例如用户从手机分享 30 张照片到电脑，中间断了。下次两台设备在线后，不应该让用户一个个找，而应该显示：

```text
上次未完成批次：30 个文件，已完成 18 个，剩余 12 个
[一键继续]
```

这个体验非常强。

## 这个优势还能反过来解释“隧道”的必要性

之前你的问题是：用户为什么要进入一个“隧道”？听起来抽象。

现在有答案了：

> 因为隧道不是房间，而是这批传输任务的容器。

两台设备加入同一个隧道，本质上是在建立一个共享任务上下文。这个上下文可以跨时间存在，不要求双方一直在线。

这就让“隧道”这个概念合理了：

* 临时传输：打开就传；
* 中断挂起：任务留在隧道；
* 再次相遇：自动恢复；
* 批量管理：同一批资源一起处理；
* 清理权限：用户决定保留、恢复、删除。

## 你可以把竞品差异总结成一句话

> PairDrop 更像一次性递东西；file-tunnel 更像一个可恢复的传输任务箱。

或者：

> 别人解决“现在发过去”，file-tunnel 解决“没发完也可控，之后还能接着发”。

这个角度比“我也支持 P2P 文件传输”强很多。

## 下一步优先级

我建议你后续围绕这个优势做三件事：

第一，**显式展示挂起任务**。不要只藏在资源浏览器里，要让用户看到“未完成、等待来源设备、可恢复”。

第二，**做批次恢复**。一次分享/一次上传形成一个 batch，支持“一键恢复本批次”。

第三，**做恢复承诺文案**。例如：

```text
文件已加入隧道。即使现在没传完，只要双方以后再次在线，仍可继续传输。
```

这句话应该出现在上传后、断线后、恢复中心里。

最终你的产品核心可以变成：

> file-tunnel 是一个跨设备、可挂起、可恢复、可管理的传输任务容器。

这个定位很准，也比单纯“即时传输工具”更有护城河。
